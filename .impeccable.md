# Design Context for Symphony Orchestrator

## Design Context

### Users

Developers and operators running Symphony locally to monitor, configure, and trust autonomous coding work. They are usually multitasking across terminals, issue trackers, repositories, and dashboards. They do not want ornamental UI; they want fast scanning, dependable state cues, and a clear sense of whether the system is healthy, live, blocked, or waiting.

Their typical context is:

- a local development machine
- multiple terminals and browser tabs open
- intermittent attention on the dashboard rather than continuous focus
- a preference for dense, high-signal interfaces over marketing-style presentation

### Brand Personality

**Precise, calm, reliable.** Symphony should feel like a serious engineering instrument: composed under pressure, operationally trustworthy, and quietly opinionated. The interface can feel distinctive, but never playful at the expense of clarity.

The product should communicate:

- technical competence
- runtime trustworthiness
- operational calm
- crisp, intentional design judgment

It should not communicate:

- consumer-SaaS softness
- novelty-first visual experimentation
- playful friendliness
- decorative motion for its own sake

### Aesthetic Direction

- **Visual tone:** Mission-control UI with restrained warmth. Information-dense, sharp, and compositional rather than soft or decorative.
- **Theme:** Dual theme support, with dark mode as the default operating environment.
- **Color system:** Copper is the brand and primary-action accent on slate neutrals. Semantic status colors stay operational: blue for queued, purple for claimed, green for running, amber for retrying, red for blocked, gray for completed.
- **Typography:** Space Grotesk for headings, Manrope for body and UI copy, IBM Plex Mono for identifiers, counters, metrics, and logs.
- **Shape language:** Mostly square or near-square edges. Small radii are acceptable for compact controls, but the overall silhouette should stay crisp and stitched, not pillowy.
- **Motion:** Subtle and purposeful. Motion should clarify live state, transitions, focus, or hierarchy, not entertain.
- **References:** Developer tools like Linear, Raycast, GitHub, and terminal-first dashboards that privilege signal, structure, and speed.
- **Anti-references:** Avoid glossy gradients, bubbly cards, playful illustrations, oversized empty states, over-rounded surfaces, or anything that makes the dashboard feel consumerized.

### Design Principles

1. **Operational clarity first** — A user should understand current state, urgency, and next action in seconds.
2. **Semantics over decoration** — Accent color is for brand and primary action; status colors are for runtime meaning.
3. **Sharp, confident composition** — Prefer crisp edges, deliberate spacing, and strong information hierarchy over soft visual comfort.
4. **High signal per pixel** — Every panel, badge, strip, and toolbar should justify its presence in a constrained workspace.
5. **Accessible by default** — Keyboard-friendly flows, durable contrast, and motion that still works in reduced-attention contexts.

## Spacing, Shape, And Motion

### Spacing

Base spacing starts at `--space-1` (4px) and extends through larger section gaps such as:

- `--space-2`: 8px
- `--space-3`: 12px
- `--space-4`: 16px
- `--space-5`: 20px
- `--space-6`: 24px
- `--space-8`: 32px
- `--space-10`: 40px
- `--space-12`: 48px
- `--space-16`: 64px
- `--space-20`: 80px

### Radius

The live implementation is sharper than typical UI systems:

- `--radius-sm`: 2px
- `--radius-md`: 2px
- `--radius-lg`: 0px
- `--radius-xl`: 0px
- `--radius-full`: 9999px

That means compact controls can have a slight softening, but major surfaces should remain visually square.

### Controls And Motion

- Control heights run from `--control-height-xs` (28px) through `--control-height-xl` (44px).
- Motion stays restrained: `--motion-instant`, `--motion-fast`, `--motion-medium`, `--motion-slow`.
- Use motion to indicate live state, focus change, reveal order, or interaction response.
- Avoid ornamental animations, bouncing entrances, or glossy transitions.

- `mc-button-ghost`
- `mc-button-secondary`
- `mc-btn`
- old teal-era “brand” assumptions

## Implementation Guidance

When designing or refactoring UI in this project:

1. Prefer existing `mc-*` primitives and tokens over page-local one-offs.
2. Use semantic status colors for runtime state, not copper.
3. Use copper for brand, active navigation, and primary action only.
4. Preserve the sharp, composed, information-dense visual language.
5. Keep dark mode parity by using tokens instead of absolute colors.
6. Update this file when the live frontend changes materially, not before.
